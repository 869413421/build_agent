# 教程黄金标准（跨会话强制基线）
# 一、核心定位（最高优先级）

本教程属于：

> 单工程 · 增量构建 · 架构演进型连载教程

定义：

1. 所有章节构建同一个工程
2. 代码只能新增或扩展，不允许推翻式重写
3. 不允许 demo 版本
4. 不允许未来删除的过渡代码
5. 每章结束必须形成真实增量闭环

若违反以上任意条目，视为架构破坏。

## 适用范围

1. `docs/tutorials/01..14` 全部章节
3. `src/agent_forge` 主线代码

---

# 二、章节核心目标

每章必须明确回答：

* 本章新增什么系统能力？
* 该能力在整体架构中处于哪个层级？
* 为什么必须在此时引入？

禁止模糊目标。

---

# 三、固定章节结构（强制）

顺序不可变：

1. 目标（本章新增能力）
2. 架构位置说明（系统全局图中的位置）
3. 前置条件
4. 实施步骤（先讲“面”，再讲“点”）
5. 运行命令
6. 增量闭环验证
7. 常见失败模式
8. 本章 DoD
9. 下一章预告（架构演进视角）

---

# 四、架构演进强约束规则

## 1️⃣ 不可推翻原则

禁止：

* 本章写简化实现，下章重写
* 本章写 stub，下章删除
* 本章写 demo，下章替换

允许：

* 本章设计可扩展结构
* 下章在原结构上扩展

---

## 2️⃣ 增量闭环强制规则

每章结束必须满足：

* 主程序可运行
* 本章新增模块已真实接入主流程
* 输出可验证新增能力生效
* 无孤立代码

---

## 3️⃣ 架构一致性约束

若新增模块：

必须说明：

* 它属于哪一层？
* 是否破坏依赖方向？
* 是否引入循环依赖风险？
* 是否影响已有接口？

---

# 五、教学节奏规则（演进型版本）

## 1️⃣ 禁止 Demo 式“最小可跑”

替换为：

> 本章增量能力验证

定义：

验证必须基于真实架构。

---

## 2️⃣ 概念必须服务于架构

每个新概念必须回答：

* 它解决架构中的什么问题？
* 为什么当前时机必须引入？
* 若延迟引入会导致什么结构问题？

---

## 3️⃣ 禁止抽象堆砌

引入概念必须：

* 首次出现一句话解释
* 紧贴代码实现
* 不写历史背景
* 不写泛理论

---

# 六、代码标准（升级版）

## 1️⃣ 代码必须为最终架构一部分

禁止：

* 占位函数
* TODO 未来删除
* 伪实现
* 简化示意版本

---

## 2️⃣ 每个核心代码块后必须包含：

### 四维工程讲解

* 设计动机
* 架构位置
* 工程取舍
* 失败模式

少一项不合格。

## 3️⃣ 大文件节选规则（强制）

允许在超大文件场景下展示“节选代码”，但必须同时满足：

* 紧邻节选代码处明确标注“本段为节选”
* 给出对应源码文件的可点击路径链接
* 明确提醒读者“需进入该文件复制完整代码后再运行”

## 4️⃣ 讲解表达规则（强制）

代码讲解必须满足以下要求（除简单导出文件如 `__init__.py` 外）：

* 通俗易懂：优先使用生活化类比或真实工程场景，不只给术语定义
* 适当举例：至少提供一个“成功链路例子”与一个“失败链路例子”
* 图文并茂：复杂流程至少补一张流程图或时序图
* 讲解深度统一：每段“主要代码”（核心流程文件）必须达到 `chain_runner.py` 讲解级别，至少包含“主流程拆解 + 成功链路例子 + 失败链路例子 + 流程图/时序图 + 工程取舍与边界”
* 增量升级：优化历史章节时只允许追加讲解，不允许删减原有内容和代码
* 自然融合：禁止“补丁式”写法（如“补充说明/修复说明/本次追加”口吻独立成块），必须把内容无痕融入主线叙事

---

# 七、文件创建强约束

## 每个“文件：path”前必须紧邻：

```bash
touch path
```

```powershell
New-Item -ItemType File path
```

规则：

* 仅创建该文件
* 不批量创建
* 路径必须真实
* 与仓库一致
* 若新增的是 Python 包目录（会被 import），必须在同一步创建并展示对应 `__init__.py`（含创建命令与完整代码）

---

## 禁止：

* 一口气创建全部目录
* 使用伪路径
* 隐式存在文件
* 在教程步骤中直接 `from ... import ...`，但未提前创建该包目录下的 `__init__.py`

---

# 八、环境控制规则

* Python 统一使用 `uv`
* 所有命令必须 Windows PowerShell 可执行
* 若复用环境，必须提供一条验证命令

---

# 九、工程真实性规则

每章必须包含：

* 至少 1 个真实失败场景
* 至少 1 个架构权衡讨论
* 至少 1 个“为什么不用另一种方案”
* 至少 1 个边界条件讨论

否则视为说明书腔。

---

# 十、质量红线（命中即重写）

出现任一条立即失败：

* 写 demo 版本
* 推翻前文结构
* 存在未来删除代码
* 主程序无法真实运行
* 无架构位置说明
* 无增量闭环验证
* 缺承上启下
* 抽象理论堆砌
* 代码与主线项目不一致，缺少或漂移

---

# 十一、自检闭环（强制执行）

输出完成后必须逐项确认：

* 是否单工程连续构建？
* 是否未推翻前文结构？
* 是否形成真实增量闭环？
* 是否无占位代码？
* 是否每个核心代码块有四维讲解？
* 是否包含真实失败模式？
* 是否承上启下？

若任一为否，必须重写。

---

# 十二、承上启下强制规则

下一章预告必须说明：

* 当前架构已具备什么能力？
* 现阶段最大结构缺口是什么？
* 下一章为何必须补这个缺口？
* 若不补会出现什么工程问题？

禁止泛泛预告。

---

# 十三、最终判定标准

本教程必须达到：

> 读者完成所有章节后，得到的是一个可持续扩展的真实工程，而不是拼装示例合集。
