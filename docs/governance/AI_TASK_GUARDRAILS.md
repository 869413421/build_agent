# AI 任务执行规则

## 强制对齐

执行任何任务前，必须先阅读：

1. `docs/governance/NORTH_STAR.md`
2. `docs/governance/AI_TASK_GUARDRAILS.md`
3. `AGENTS.md`
4. `docs/governance/PROJECT_STATUS.md`
5. `docs/tutorials/_TUTORIAL_GOLD_STANDARD.md`

## 硬约束

1. 所有协作沟通与教程默认使用中文。
2. 全仓文件统一为 UTF-8（无 BOM），禁止混用本地编码。
3. 框架层设计必须保持领域无关，不绑定单一业务场景字段。
4. 业务场景约束（如法律免责声明）只能出现在应用层，不应固化在通用协议层。
5. 框架目标是生产可用，不接受“仅演示可跑”的玩具化实现。
6. Engine 主循环必须实现 `plan -> act -> observe -> reflect -> update -> finish`，不得省略 reflect。
7. 新增能力必须通过接口扩展，不得破坏既有公开契约（向后兼容优先）。
8. 每次只实现一个框架组件，不允许跨组件大步提交。
9. 每步必须同时提交：代码 + 测试 + 教程。
10. 代码必须包含清晰、明确的关键注释，解释意图与边界条件。
11. 未经用户审核“通过”，禁止进入下一步。
12. 注释风格必须强调“为什么这样设计”，不能只描述代码表面行为。
13. 任何代码结构或注释风格变化，必须同步更新对应教程章节。
14. 执行 Python 相关命令时优先使用 `uv`（如 `uv run ...`、`uv sync --dev`）。
15. 新增或修改教程时，必须以 `docs/tutorials/_CHAPTER_TEMPLATE.md` 为骨架，并满足黄金标准。
16. 生产要求默认开启：并发隔离、版本追踪、故障域控制、隐私脱敏、可回放、评测门禁。
17. 教程中的代码片段必须是“可直接运行的完整代码”，不得只给伪代码或省略关键实现。
18. 教程中提到的代码文件路径必须使用可点击跳转的相对路径链接，并与仓库真实文件一致。
19. 教程中禁止使用“完整代码见仓库”替代关键内容，核心示例与测试代码必须在文内完整展示。
20. 教程必须包含“代码讲解”内容，且要解释为什么这样设计、替代方案取舍、边界条件与失败模式。
21. 教程结构必须遵循“先面后点”：先独立讲清主流程/全局链路，再展开关键实现细节。
22. 业务代码中的每一个方法/函数必须有注释（docstring），禁止遗漏私有方法与工具函数。
23. 方法注释必须包含参数级说明（`Args`）以及返回值说明（`Returns`）；存在异常分支时必须补 `Raises`。

## 质量门禁

1. 功能变更必须同步更新测试。
2. API/Schema 与实现必须保持一致。
3. 设计变化必须记录到 `docs/architecture/`。
4. 会话结束前必须更新 `docs/governance/PROJECT_STATUS.md` 与 `docs/governance/HANDOFF_CHECKLIST.md`。
5. 性能与稳定性必须纳入验收：至少提交 P95 延迟、错误率、重试率与吞吐基线。
6. 任何会进入日志/trace/audit 的敏感字段必须明确脱敏策略。

## 完成定义（DoD）

1. 本地可运行。
2. 对应测试通过。
3. 对应教程可复现。
4. 安全边界未被破坏。
